# 待优化清单

本文档只记录后续优化建议和检查发现的维护风险，不修改当前源代码行为。清单按“性能优化型”和“功能增减型”归类，优先级使用 P0/P1/P2 表示：P0 为发布或稳定性高风险，P1 为近期值得做，P2 为后续体验增强。

## 性能优化型

### P0. 清理 `public/style.css` 的重复覆盖块

当前 `public/style.css` 存在多段类似“final overrides / EOF override”的重复规则，文件体积已经明显偏大，且样式来源难以追踪。建议拆分为：

```text
public/styles/base.css
public/styles/overlay.css
public/styles/dashboard.css
public/styles/editor.css
```

如果暂时不拆文件，也应先按区域整理章节，并删除过期覆盖块，避免后续样式继续靠末尾覆盖堆叠。

### P1. 降低 OBS 源队列渲染和滚动测量成本

`public/app.js` 在每次 `queue:state` 时会重建整个队列 DOM，并触发队列项与队列容器的滚动测量。当前队列上限不高，问题不严重；但如果后续提高队列容量、增加封面或更多状态，OBS 源会更容易卡顿。

建议：

- 按 `requestId` 做增量更新，避免完整 `innerHTML`/DOM 重建。
- 只在队列内容或外观尺寸变化时重新测量滚动。
- 给滚动测量增加批处理或去抖，减少连续 Socket.IO 事件造成的 layout thrashing。

### P1. 给网易云搜索与播放地址解析增加可控并发和短期缓存

当前每条有效点歌都会执行搜索、可用性检查、播放地址获取，且 `resolveSong()` 对搜索结果逐个串行尝试。直播间弹幕集中出现时，外部 API 会成为主要延迟点。

建议：

- 对相同 keyword / songId 增加短期内存缓存。
- 对同一时间的解析任务设置并发上限，避免外部 API 抖动拖垮主进程。
- 对 `check_music` 和 `song_url_v1` 的串行尝试做限量并发或失败快速退出。

### P1. 音频下载增加超时、取消和临时文件清理

`audioCache.download()` 使用 `fetch()` 拉取远端音频，但没有显式超时或取消控制；下载失败时 `.tmp` 文件可能残留，预热失败也被静默吞掉。建议：

- 使用 `AbortController` 对音频下载设置超时。
- 下载失败或中断时清理对应 `.tmp` 文件。
- 预热失败时向控制台发出可观测日志，便于排查卡顿和 502。

### P1. 限制内存状态增长

`queue.history` 和 `lastRequestByUser` 当前没有上限清理。`publicState()` 只返回最近 20 条历史，但内存中的历史数组仍会持续增长；长时间直播后会产生不必要的内存占用。

建议：

- 为历史记录设置最大保留条数。
- 为用户冷却 Map 增加 TTL 清理。
- 在控制台展示清理后的统计信息，而不是无界保留运行期数据。

### P1. 减少高频 B 站状态广播

`src/bilibili.js` 对非弹幕消息、心跳和每条弹幕都会更新并广播 `bilibili:status`。控制台有用，但 OBS 源并不总需要完整状态。建议：

- 对心跳和非弹幕 `MESSAGE` 的状态推送做节流。
- 将弹幕日志和连接健康状态拆成不同事件频率。
- 新客户端连接时仍发送完整快照，运行期只发送必要差量。

### P2. 提取前后端共享外观 schema

外观默认值和字段读取逻辑同时存在于 `src/appearance.js` 和 `public/dashboard.js`。这会增加维护成本，也容易出现字段遗漏。建议将默认值、范围、字段类型抽成可共享的 JSON/schema，由服务端返回给前端渲染编辑器。

### P2. 用构建流程分离静态资源

当前前端是直接加载单个 `style.css` 和手写 JS，适合小规模项目。后续若继续扩展控制台，建议增加轻量构建流程，输出按页面拆分的 CSS/JS，并带 hash 版本，减少 OBS 源加载与缓存失效成本。

## 功能增减型

### P0. 移除 `.env.example` 中的真实凭据

`.env.example` 当前包含完整 B 站 Cookie、`DedeUserID`、`SESSDATA`、`bili_jct` 等登录凭据。开源或共享前必须替换为占位值：

```text
BILI_COOKIE=replace_me
NCM_COOKIE=
```

真实 `.env` 已在 `.gitignore` 中，但示例文件同样不能保留真实账号信息。

### P1. 修复点歌冷却的扣减时机

`handleDanmaku()` 会在解析到点歌命令后立刻调用 `canUserRequest()`，即使后续搜索失败、歌曲不可播、队列已满或重复歌曲，用户也会进入冷却。建议把冷却提交改为“通过校验并准备入队后”再记录，或者拆成 `peekCooldown()` 与 `commitCooldown()` 两步。

### P1. 修复队列上限统计口径

`addSong()` 只检查候选队列长度，未把当前播放歌曲计入上限。若 `MAX_QUEUE_SIZE=30`，实际可能出现 1 首当前播放 + 30 首候选。需要明确配置语义：

- 如果上限表示候选队列，文档和 UI 应写成“候选队列上限”。
- 如果上限表示总点歌池，应改为 `queue.length + (current ? 1 : 0)`。

### P1. 处理 OBS 源错过 `player:play` 事件后的同步问题

`public/app.js` 在收到 `queue:state` 且 `activeRequestId` 不同的时候只更新标识，不会主动播放新歌。大多数情况下 `player:play` 会随后或此前到达，但如果 OBS 源重连、事件乱序或刷新时错过 `player:play`，可能只显示当前歌曲而不播放。

建议在 `queue:state` 检测到当前歌曲变化时，根据播放状态决定是否调用 `playSong(state.current)`，并避免重复 `load()` 当前音频。

### P1. 修复 B 站切房失败后的状态回滚

`startBilibili()` 在连接成功前就更新 `config.roomId`、`process.env.BILI_ROOM_ID` 和 `bilibiliStatus`。如果 `connectBilibili()` 失败，内存房间号会变成新房间，但 `.env` 不会写入，旧连接也已关闭，控制台会进入不一致状态。

建议连接成功后再提交房间号，或在失败时回滚 `config.roomId` 并恢复旧连接/明确显示失败状态。

### P1. 明确手动点歌接口与控制台入口

服务端已有 `POST /api/request`，但架构文档此前未列出，控制台也没有手动搜索/插入入口。可以二选一：

- 增加控制台手动点歌 UI，供主播主动插入歌曲。
- 如果暂不支持，把接口标记为调试接口，避免误以为是完整功能。

### P1. 补齐自动重连策略

当前 close/error 只更新连接状态，不会自动重连。建议为 B 站弹幕连接增加指数退避重连，并在切房时取消旧 session 的后续事件。控制台可显示重连次数、下次重连时间和最后错误。

### P1. 细化网易云播放失败原因

`resolveSong()` 会跳过不可播放结果，最终只返回 `null`。建议把失败原因细化为：

- 无搜索结果
- 版权不可用
- 需要会员但未登录
- 登录后仍无权限
- 获取播放地址超时

这样控制台和 OBS 播报可以给主播更明确的反馈。

### P1. 修复控制台设置存储命名不一致

架构文档已按代码更正为 `.cache/dashboard.json`。如果希望文件名更直观，可以后续把代码迁移到 `.cache/dashboard-settings.json`，并兼容读取旧文件。

### P1. 增加更完整的测试

当前 `npm test` 是外部 API 冒烟测试。建议补充不依赖外网的单元和集成测试：

- 弹幕文本解析测试
- 队列去重、冷却、上限测试
- 外观配置归一化测试
- 音频缓存 Range 请求测试
- 控制台保存/读取配置测试
- `queue:state` 与 `player:play` 事件顺序测试

### P2. 增加音频缓存持久化索引

当前缓存注册表在内存中，服务重启后旧缓存文件仍在，但旧 `requestId` 不再可播放。可以考虑持久化缓存索引，或者启动时扫描 `.cache/audio` 恢复可引用条目。

### P2. 增加导入/导出外观方案

目前支持保存配置、读取配置、恢复默认。后续可以增加：

- 导出 JSON
- 导入 JSON
- 多套预设命名
- 预设缩略图

### P2. 增加队列管理高级功能

可以考虑：

- 拖拽调整候选歌曲顺序
- 置顶某首歌
- 拉黑关键词
- 拉黑用户
- 主播手动插队

### P2. 增加弹幕反馈

如果未来需要自动回复弹幕，可以加入点歌成功、失败原因、当前队列位置等反馈。注意这会涉及 B 站发送弹幕接口和风控，建议谨慎实现。

### P2. 控制台交互增强

建议后续增加：

- 外观编辑器撤销上一步
- 颜色选择器常用色板
- 毛玻璃“深色直播 / 浅色直播 / 高对比”预设
- 控制台日志按“弹幕 / 点歌 / 系统”分组过滤
- 本机字体读取兼容提示和手动输入字体名

## 开源发布前检查

- 确认 `.env`、`.cache/`、日志文件不提交。
- 确认 `.env.example` 没有真实 Cookie、用户 ID 或其他登录凭据。
- 补充截图或 GIF。
- 标注 Node.js 推荐版本。
- 标注 `@neteasecloudmusicapienhanced/api` 的使用原因。
- 标注网易云登录和会员歌曲播放的限制。
- 增加贡献指南和 issue 模板。
